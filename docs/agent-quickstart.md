# Agent quickstart

A copy-pasteable walkthrough of the `datuplet` CLI's agent loop (RFC 027):
discover a component's config shape, compose a pipeline doc, validate it
*before* persisting anything, upsert it, trigger a run and wait for it, then
inspect the result in storage. Every command below is real — checked against
`cmd/datuplet/{main,pipeline,components,trigger,storage}.go` — and every
output shape is either lifted from the source or, for the validate finding,
generated by actually running the validator against the exact YAML shown
(see the callout in Step 3).

This walkthrough assumes **no prior `~/.datuplet` state** — the target
audience is a script or agent starting from a bare environment.

## Prerequisites

Pick one of two auth paths (RFC 027 §7):

**A. Headless (env vars only) — no files, no login flow:**

```bash
export DATUPLET_REMOTE=https://pipeline-api.example.com
export DATUPLET_API_TOKEN=<your cli-api bearer token>
```

With both set, `loadRemoteArgs` never touches `~/.datuplet` at all
(`cmd/datuplet/remote.go`'s "headless fast path"). One consequence: there is
no on-disk project list to auto-default from, so **every project-scoped
subcommand** (`pipeline`, `trigger`, `storage` — but not `components`, which
isn't project-scoped) **needs an explicit `--project <project-uuid>`** — the
literal Datuplet project UUID, not a human-readable name. Get the UUID from
whoever provisioned your project access.

**B. Explicit non-interactive login:**

```bash
echo "$DATUPLET_PASSWORD" | datuplet login --remote https://pipeline-api.example.com \
  --email you@example.com --password-stdin
```

This writes `~/.datuplet/token`, `~/.datuplet/api-token`, and
`~/.datuplet/cluster.json` (0600). After this, `--remote`/`--token-file` can
be omitted, and `--project` takes the human-readable project *name* — it
auto-defaults if you have exactly one project.

The rest of this walkthrough uses path A (env vars); substitute your own
`--project` value throughout. Set it once:

```bash
export PROJECT=3f9f9d2e-6b7a-4c1b-9e2f-6a2b7c9d1e10   # your project UUID
```

## Step 1 — Learn the component catalog

```bash
datuplet components list
```

```
NAME                     DISPLAY                  DEFAULT      IO                 DEPRECATED
data-generator           Data Generator           v0.9.1       none/required      false
sql-transform            SQL Transform            v0.9.1       required/required  false
finnhub-extractor        Finnhub Extractor        v0.9.1       none/required      false
http-json-extractor      HTTP JSON Extractor      v0.9.1       none/required      false
pandas-transform         Pandas Transform         v0.9.1       required/required  false
stdout-writer            Stdout Writer            v0.9.1       required/none      false
```

(`--json` prints the server's raw JSON array instead — same fields, machine
readable.)

Then pull the exact config shape for the component you want to use, so you
compose valid `config:` blocks instead of guessing:

```bash
datuplet components get data-generator --schema
```

This prints the resolved version's `configSchema` **verbatim** (nothing
else — no name/version banner, just the JSON, spec §7):

and each table's `random` mode requires **both** `schema` and `limit`.

## Step 2 — Compose the pipeline doc

Save this as `events-etl.yaml` — the spec §3 example, verbatim:

```yaml
name: events-etl          # optional in body; must equal the URL/CLI name when present
description: Generate demo events and aggregate revenue per country.  # optional
gateway:                  # optional, GatewayConfig unchanged (chunkSize, bufferSize, …)
  chunkSize: 33554432
stages:                   # sequential; components within a stage run in parallel
  - name: generate
    components:
      - name: gen                     # instance name, unique across the pipeline
        component: data-generator     # registry reference
        version: 0.9.1                # optional → registry default resolution
        config:                       # validated against the version's configSchema
          tables:
            - name: events
              rowInsertSpeed: 1000
              random:
                schema: { id: uuid, created: timestamp, amount: double, country: string }
                limit: { rowsCount: 100000 }
        outputs:
          defaultBucket: raw          # dynamic mode: component names tables at runtime
          defaultWriteMode: APPEND
  - name: transform
    components:
      - name: daily-summary
        component: sql-transform
        inputs:
          tables:
            - { bucket: raw, table: events }
        config:
          sql: |
            CREATE TABLE daily_summary AS
            SELECT country, count(*) AS events, sum(amount) AS revenue
            FROM events GROUP BY country;
        outputs:
          tables:                     # explicit mode: one mapping per table
            - { name: daily_summary, bucket: curated, writeMode: FULL_LOAD }
```

## Step 3 — Validate before persisting

Validating first means a bad doc never gets written into a project's
pipeline store — and it gives you a JSON findings shape + exit code you can
script against without ever making a mutating call (spec §7).

**Realistic failure first.** Say you fat-fingered the `gen` component's
config and dropped the `limit` block (an easy mistake — `random` mode
requires **both** `schema` and `limit`, per Step 1's schema read):

```yaml
              random:
                schema: { id: uuid, created: timestamp, amount: double, country: string }
                # limit block missing
```

```bash
datuplet pipeline validate -f events-etl.yaml --project "$PROJECT"
echo "exit=$?"
```

```
SEVERITY PATH                                     MESSAGE
ERROR    stages[0].components[0].config.tables[0].random missing property 'limit'
exit=1
```

> The finding's path and message above are not invented — they were produced
> by actually running `validate.ValidateConfig` against
> `components/data-generator/schema.json` with the `limit` block removed
> from an otherwise-identical `random` config, during authoring of this doc.

`--json` gives you the same finding as structured output, for scripting:

```bash
datuplet pipeline validate -f events-etl.yaml --project "$PROJECT" --json
```

```json
{"findings":[{"path":"stages[0].components[0].config.tables[0].random","message":"missing property 'limit'","severity":"error"}]}
```

**The fix**: put the `limit` block back (the full doc from Step 2 already
has it). Re-run:

```bash
datuplet pipeline validate -f events-etl.yaml --project "$PROJECT"
echo "exit=$?"
```

```
no findings
exit=0
```

Exit-code contract (spec §7, `cmd/datuplet/pipeline.go`'s `runPipelineValidate`):
`0` = no error-severity finding (warnings still print), `1` = at least one
error-severity finding, `2+` = the validate request itself failed
(transport/HTTP problem, not "the pipeline is invalid").

## Step 4 — Persist it

```bash
datuplet pipeline put -f events-etl.yaml --project "$PROJECT"
```

```
pipeline "events-etl" upserted in project ""
```

(The trailing `""` is expected in pure env-var headless mode: there's no
on-disk project list to resolve a display name from, so `ProjectName` is
blank — the upsert itself still lands in the correct project, addressed by
the `--project` UUID. Logging in via `datuplet login` instead resolves and
prints the real project name here.)

`put` takes the positional pipeline name from the doc's own `name:` field
when omitted, so `datuplet pipeline put -f events-etl.yaml` (no name arg) is
equivalent here — the doc's `name: events-etl` supplies it.

## Step 5 — Trigger and wait

```bash
datuplet trigger --project "$PROJECT" --wait --timeout 10m --json events-etl
```

(`--wait` polls the run every 2s until it reaches a terminal phase, up to
`--timeout` — default `1h` — cancelling cluster-side on expiry; the pipeline
name is positional and must come **after** every flag, since Go's `flag`
package stops parsing at the first non-flag argument.)

```json
{
  "id": "b6f2b8b0-8e35-4b0a-9f1c-2a7d4e8c9a11",
  "project_id": "3f9f9d2e-6b7a-4c1b-9e2f-6a2b7c9d1e10",
  "pipeline_id": "a1c3...",
  "pipeline_name": "events-etl",
  "phase": "Succeeded",
  "created_at": "2026-07-20T10:00:00Z",
  "ended_at": "2026-07-20T10:00:42Z",
  "wallclock_seconds": 42
}
```

The process exit code mirrors the phase: `0` for `Succeeded`, `1` for
anything else (`FailedUser`, `FailedApplication`, `Cancelled`, `Expired`).
Drop `--json` for a one-line human summary instead:
`run <id> (pipeline=events-etl) phase=Succeeded wallclock=42s`.

## Step 6 — Verify the output

```bash
datuplet storage sample curated.daily_summary --project "$PROJECT"
```

(`storage sample` takes a single `<namespace>.<table>` reference — here
`curated.daily_summary`, matching the `transform` stage's output mapping
from Step 2. Add `--rows N` to change the preview size; omit it for the
server default.)

```json
{
  "columns": [
    { "name": "country", "type": "VARCHAR" },
    { "name": "events", "type": "BIGINT" },
    { "name": "revenue", "type": "DOUBLE" }
  ],
  "rows": [
    ["US", 41213, 812345.67],
    ["DE", 18422, 305114.02]
  ],
  "truncated": false
}
```

Other `storage` subcommands you'll want alongside `sample`: `tables` (list
every table in the project), `info` / `schema` / `history` — same
`<ns>.<table>` argument, same `--project` requirement.

## Full loop, back to back

```bash
export DATUPLET_REMOTE=https://pipeline-api.example.com
export DATUPLET_API_TOKEN=<your cli-api bearer token>
export PROJECT=<your project uuid>

datuplet components get data-generator --schema        # learn the config shape
datuplet pipeline validate -f events-etl.yaml --project "$PROJECT"   # findings + exit code
datuplet pipeline put -f events-etl.yaml --project "$PROJECT"
datuplet trigger --project "$PROJECT" --wait --json events-etl
datuplet storage sample curated.daily_summary --project "$PROJECT"  # verify output
```
